DROP VIEW viewname

a view is updatable if we dont have DISTINCT, aggregate funcs, groupby or union in the select statement... we can update rows or delete rows etc. like a regular table

while updating a row in a view, some rows might disappear... to stop updating rows that will result in disappearance of a row, CREATE VIEW viewname AS select statement WITH CHECK OPTION 

e.g. if views has WHERE dept='HR' in it, trying inserting dept='IT' will result in vanishing row

views can reduce impact of changes, restrict access to the data

transactions = statements that represent single unit of work

acid = atomicity, consistency, isolation (diff transactions are isolated), durability (perm changes)

```

START TRANSACTION

insert into order;

insert into order\_items;

COMMIT;

each statement in SQL is by default a transaction

concurrency=multiple users accessing some data at same time... to handle this, mysql puts a lock on it

lost updates = later transaction overwrite previous transaction

dirty reads = A update but doesnt commit yet but B reads old value... to fix this we use isolation level READ COMMITED so that it only read commited data

non-repeating inconsistent reads = A reads 10 first and B updates it to 0 and subquery of A reads 0 which is wrong... to fix this we use isolation level REPEATABLE READ to read same data everywhere in a transaction

phantom reads = A selects some data but a new entry appears after that read... to fix this we use isolation level SERIALIZABLE, this makes sure transaction is aware of all changes in the data... this can hurt performance

SET SESSION/GLOBAL TRANSACTION ISLOATION LEVEL SERIALIZABLE;

deadlock = transactions keep waiting for each other and never release a lock

in following statements deadlock can occur: 

START TRANSACTION UPDATE orders; UPDATE customers; COMMIT;

START TRANSACTION UPDATE customers; UPDATE orders; COMMIT;

normalization = to remove redundant or duplicated data

1nf = each self have single value and no repeated columns (tag1,tag2 etc. instead make a tags table)

2nf = 1nf and each table describe one entity and every column describe that entity e.g. order id storing customer name is not good, instead use customer id so that name is not repeated

3nf = 2nf and a column in table should not be derived from other columns

build model on current requirements, not very far future requirements

composite indexes: CREATE INDEX idx\_state\_points ON customers (state,points);

my sql allows 16 columns max in composite index.

rule1: put the most frequent columns must come first - rule2: high cardinality must come first

if we have WHERE state='something' OR points=1000, index might not work, instead create seperate indexes for both state and points and union two select statements... similarly performing arithematic operations such as points+10=2010 can result in low performance, instead do points=2000



we can use indexes for sorting data.

if we have composite index (a,b) we can sort a alone, a and b, a DESC and b DESC. But we cant sort b only cheaply.

covering index =  an index which contains all the columns required to satisfy a query... index automatically always include the PK

do not create duplicate indexes (a,b) and (a,b)

do not create redundant indexes like (a,b) and a or (b,a) and b
